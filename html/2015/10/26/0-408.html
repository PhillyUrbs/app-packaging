Note
If you install from a source other than the deploy folder, the application will not automatically prompt the user for an update when the server is updated.

Advantage:

We control our workstation updates and have access to our deployment reports and tools to show that the job is done.

You can also install via the System account and don't have to provide users permissions to allow for the update to install without Admin rights to the workstation.

Note
The installation options can be found in the owner's manual, but that's if the user gives you the owner's manual. :-) This is how you can deploy out Raiser's Edge with command-line switches

REM Install Raiser's Edge
\\raiser\deploy\setup.exe /S/v/qn /V"/qn INSTALLLEVEL=60 REINSTALLMODE=veums"
rem Icon Cleanup
del "C:\Documents and Settings\All Users\Desktop\RE Release Notes.lnk" /q /f
del "C:\Documents and Settings\All Users\Desktop\The Raiser's Edge.lnk" /q /f
REM Reboot
shutdown -r -t 0 -f

Command Line
To silently install The Raiser's Edge and its prerequisites, enter this command string into your script or command line:

"setup.exe" /S/v/qn /V" /qn INSTALLLEVEL=60 REINSTALLMODE=veums"

To install The Raiser's Edge without the sample database, remove INSTALLLEVEL=60 from the command.

To silently install The Raiser's Edge only, enter this command string into your script or command line to begin a silent installation:

"The Raisers Edge.msi" /qn/I" INSTALLLEVEL=60 REINSTALLMODE=veums".

To install The Raiser's Edge without the sample database, remove INSTALLLEVEL=60 from the command.

Note
This problem gave me a few headaches. got this from vendor
----------------------------------------------------------
Description
When logging into The Raiser's Edge on a workstation, a user may receive the following error: Error: Access to The Raiser's Edge install directory and subdirectories has been denied. User rights should be checked.
Environment


Answer
Try each of the following steps in order until the issue is resolved:
1. Make the user a Local Administrator or set the appropriate NTFS permissions
2. Manually register the DAO360.DLL file. This file is typically located in the C:\Program Files\Common Files\Microsoft Shared\DAO\ directory
Note: If unable to register the DAO360.dll file, then open Add/Remove Programs and ensure Microsoft .NET Framework 2.0 is installed. If there is only a listing for Microsoft .NET Framework 2.0 Service Pack 1, see BB463670
3. If the database is using a full instance of SQL Server 2005, ensure that the REUSER SQL account has permission to view server state
4. Uninstall and reinstall the workstation.

Note

We actually had prepared to do it via Kace. We followed the instructions in the RE documentation, but what ended up happening is that the update process was triggered when the user attempted to launch the program after we upgraded the database. Not only that, but even after that upgrade, there was an additional patch that needed to be applied (albeit in the same manner), so because our user list was small we ended up just logging on to each machine as local admin and running it manually. 

It appears that RE really tries to make it easier to deploy, but in doing so, makes using tools like Kace more difficult.
